# piki-core

UI-agnostic library crate. **Must NOT depend on `crates/tui` or `crates/api-client`.**

## Modules

- `chat.rs` — `ChatMessage`, `ChatRole` (System, User, Assistant, Tool), `ChatConfig`, `ChatServerType`, `ToolDefinition`, `ToolCall`, `ToolResult` — domain types for the global AI chat panel with tool-use support. `ChatServerType` enum (Ollama, LlamaCpp) selects which server backend to use. `ChatMessage` has optional `tool_calls` and `tool_call_id` fields for agentic tool-use flow. Used by both TUI and Desktop frontends.
- `domain.rs` — `AIProvider` (with `Custom(String)` variant), `WorkspaceInfo`, `WorkspaceType` and other shared types. Re-exported from `lib.rs`.
- `providers.rs` — `ProviderConfig`, `PromptFormat`, `ProviderManager` — user-configurable providers loaded from `providers.toml`. Manages custom provider binaries, args, prompt format, and agent directories.
- `storage/` — Trait-based storage layer. Traits in `mod.rs` (`WorkspaceStorage`, `ApiHistoryStorage`, `UiPrefsStorage`, `AgentProfileStorage`), SQLite backend in `sqlite.rs`, legacy JSON in `json.rs`.
- `workspace/` — Git worktree CRUD (`WorkspaceManager`), `FileWatcher` (notify crate), config persistence.
- `pty/` — `PtySession` wrapping `portable-pty` with `spawn_blocking` for non-blocking reads, `vt100::Parser` for terminal state. The reader optionally feeds bytes through `shell_integration::parser::OscParser` and updates per-session `ShellSession` state (cwd, last command exit code).
- `shell_integration/` — Init scripts (zsh, bash) embedded via `include_str!`, `OscParser` (streaming OSC 133/7 state machine), `install` module that detects shell family and prepares env vars + extra args (`ZDOTDIR` / `--rcfile`) so the user's shell sources our bridge before its real dotfiles. Linux/macOS only.
- `cli_agent/` — Structured agent lifecycle channel: `CliAgentEvent`/`CliAgentState` (the shared protocol both frontends render), `sock` (per-tab FIFO reader), and one install module per agent whose hooks we bridge. `bridge_for_command()` maps a provider's binary to its bridge and is the **only** place that decides whether a tab gets the structured channel — never re-hardcode `cmd == "claude"` at a spawn site. `install` (Claude Code) passes a generated hook config via `claude --settings`, so it's per-spawn and touches nothing of the user's. `install_antigravity` (`agy`) has no such flag: agy only discovers hooks from a plugin, so piki writes one shared, inert-without-`PIKI_CLI_AGENT` plugin into agy's own root (`~/.gemini/config/plugins/piki-multi-bridge/`) and lets the per-tab FIFO path ride the env instead. Adding a third agent = a new `install_*` module + an `AgentBridge` variant; the spawn sites and the UI need no changes. E2E (ignored by default, drives the real binary): `crates/core/tests/cli_agent_antigravity.rs`.
- `agent_state_detect.rs` — Passive status classification for providers with no hook bridge (currently just Codex): matches a provider's OSC window-title spinner glyphs and known blocking-prompt text against a static `StateManifest` instead of an in-band protocol. Writes into the same `ShellTabState.cli_agent` field the hook bridges populate, so the Agents pane needs zero provider-specific rendering logic.
- `idle_watcher.rs` — Per-tab `IdleWatcher` for provider tabs (Custom providers). Replaces the inline byte-poll heuristic that used to live in the TUI's event loop; now shared by TUI and Desktop. Threshold is per-provider (`ProviderConfig.idle_threshold_secs`). Steps aside automatically for tabs whose `cli_agent` channel has reported, so a bridged agent never double-fires.
- `shell_env.rs` — `user_login_env()` (cached) captures the user's shell environment for GUI launches (.app/.desktop) that inherit a stripped env. **Fast path:** when stdin is a TTY (the normal TUI case) it returns `std::env::vars()` directly — spawning an interactive login shell (`-l -i -c`) to re-capture the env costs ~0.5–1s at startup (it sources the whole `~/.zshrc`) and is redundant when we were launched from a terminal. GUI launches have no TTY and take the shell round-trip. Force the round-trip with `PIKI_FORCE_LOGIN_ENV=1`.
- `git.rs` — Low-level git helpers via `tokio::process::Command`.
- `github/` — `gh` CLI wrappers for PR operations and unified diff parsing. `mod.rs` has the original per-checkout API (`get_pr_for_branch`/`get_pr_files`/`submit_review`, all keyed off a `current_dir` checkout) plus number-explicit variants that work against a detached-HEAD worktree (`get_pr_view`, `get_pr_files_by_number`, `submit_review_by_number`) — TUI's ad-hoc review flow uses the latter; Desktop still uses the former. `pr_list.rs` — `list_relevant_prs()` categorizes PRs relevant to the current `gh` user (authored / interacted-with / review-requested-pending) via four parallel `gh search prs` calls, with an exact "was I asked to review" check against the issue timeline API for already-interacted PRs. `checkout.rs` — `ReviewCheckoutManager::ensure_pr_checkout()` does an idempotent checkout of a PR into an app-managed directory (one shared base clone per repo + one `git worktree` per PR): reuses as-is if the local HEAD already matches the PR's remote head and the tree is clean, fetches+resets in place if stale/dirty, clones fresh only if no usable worktree exists yet (`plan_checkout()` is the pure decision function). Reused by `WorkspaceManager::create_review_workspace()` to turn a resolved checkout into an ephemeral `WorkspaceInfo` (`workspace_type = Simple`, `ephemeral = true`) — see `paths.rs::review_checkouts_dir()` and `domain.rs::WorkspaceInfo.ephemeral`.
- `paths.rs` — `DataPaths` struct centralizing all directory paths.
- `sysinfo.rs` / `preflight.rs` — Runtime environment checks.

## Conventions

- Error handling: `anyhow::Result` for application errors.
- Thread safety: `parking_lot::Mutex<Connection>` for SQLite, traits are `Send + Sync`.
- Schema changes: add migration in `sqlite.rs`, bump version constant.
- Storage traits: add new trait in `mod.rs`, implement in `sqlite.rs`, wire through `AppStorage`.
- All public types that cross crate boundaries go in `domain.rs`.
